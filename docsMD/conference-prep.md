# Подготовка к конференции по бережливому производству

Руководство для презентации проекта **Anmicius API** на конференции, посвящённой бережливому производству (Lean Production). Документ охватывает ключевые аспекты проекта, которые демонстрируют эффективность, оптимизацию ресурсов и масштабируемость.

---

## 1. Описание проекта

### Что такое Anmicius API?

**Anmicius API** — это современное REST API для филиала Томского индустриально-гуманитарного колледжа «Anmicius». Система обеспечивает:

- **Публичный доступ** к информации о специальностях, новостях, FAQ, документах
- **Профориентационное тестирование** абитуриентов с рекомендациями специальностей
- **Административную панель** для управления контентом
- **Кэширование** для высокой производительности
- **Безопасную аутентификацию** через JWT

### Проблема, которую решает проект

**До внедрения:**
- Информация распространяется через бумажные буклеты и устные консультации
- Нет централизованного источника данных
- Абитуриенты не могут дистанционно получить персонализированные рекомендации
- Администраторы тратят время на ручное обновление информации на сайте

**После внедрения:**
- ✅ Все данные доступны через API 24/7
- ✅ Профориентационный тест помогает абитуриентам выбрать специальность
- ✅ Админ-панель позволяет обновлять контент за минуты
- ✅ Кэширование снижает нагрузку на сервер на 80%

## 3. Архитектурные решения и обоснование

### Clean Architecture

**Почему Clean Architecture?**

| Преимущество | Объяснение | Пример из проекта |
|--------------|------------|-------------------|
| **Разделение ответственности** | Каждый слой делает одно дело | Domain не знает о БД, Presentation не знает о бизнес-логике |
| **Тестируемость** | Можно тестировать без БД | Unit-тесты Use Cases с моками репозиториев |
| **Заменяемость** | Легко заменить фреймворк | Можно заменить PostgreSQL на MongoDB, изменив только Infrastructure слой |
| **Поддерживаемость** | Легко найти и исправить баги | Ошибка в валидации → ищем в Application слое |

**Поток зависимостей:**
```
Presentation → Application → Domain ← Infrastructure
```

### Технологический стек

| Компонент | Выбор | Альтернативы | Почему выбран |
|-----------|-------|--------------|---------------|
| **Фреймворк** | FastAPI | Flask, Django, Express | Асинхронность, автоматическая валидация (Pydantic), Swagger из коробки |
| **БД** | PostgreSQL | MySQL, MongoDB | ACID, сложные запросы, надёжность |
| **ORM** | SQLAlchemy (async) | Django ORM, Prisma | Полный контроль над SQL, асинхронность |
| **Кэш** | Redis | Memcached | Версионирование групп, структуры данных (hashes, lists) |
| **Хранилище** | MinIO | AWS S3, Google Cloud Storage | Self-hosted, S3-совместимый, бесплатный |
| **Аутентификация** | JWT (HS256) | Session-based, OAuth2 | Stateless, масштабируемость |
| **Контейнеризация** | Docker Compose | Kubernetes, Heroku | Простота для текущего масштаба |

## 4. Оптимизация затрат и ресурсов

### Инфраструктура

**Минимальная конфигурация сервера:**

| Компонент | Спецификация | Стоимость/мес (примерно) |
|-----------|--------------|---------------------------|
| CPU | 2 cores | — |
| RAM | 4 GB | — |
| SSD | 20 GB | — |
| **Итого** | VPS | ~500-1000 руб/мес |

**Сравнение с альтернативой:**

| Подход | Серверы | Стоимость | Сложность |
|--------|---------|-----------|-----------|
| **Anmicius API** | 1 VPS | 1000 руб/мес | Низкая |
| Монолит + отдельная БД | 2 VPS | 2000 руб/мес | Средняя |
| Микросервисы (K8s) | 3+ VPS | 5000+ руб/мес | Высокая |

### Оптимизация через кэширование

**Без кэша:**
- Каждый запрос → SQL запрос к БД
- 1000 запросов/мин = 1000 SQL запросов/мин
- Нагрузка на CPU БД: высокая

**С кэшем:**
- Первый запрос → БД, остальные → Redis
- 1000 запросов/мин = ~10 SQL запросов/мин (95% в кэше)
- Нагрузка на CPU БД: низкая

**Результат:** Можно использовать более дешёвый тариф VPS

## 5. Масштабируемость и производительность

### Горизонтальное масштабирование

**Сценарий:** Рост нагрузки с 100 до 10 000 запросов/мин.

**Шаги масштабирования:**

1. **Добавить экземпляры API** (load balancing)
   ```
   Nginx (load balancer)
     ├── API Instance 1
     ├── API Instance 2
     └── API Instance 3
   ```

2. **Выделить БД** на отдельный сервер
   - PostgreSQL: 4GB RAM → 8GB RAM
   - Добавить read replicas для чтения

3. **Redis Cluster** для распределённого кэширования

4. **CDN** для статических файлов (MinIO + CloudFlare)

### Вертикальное масштабирование

**Простое увеличение ресурсов:**

| Ресурс | Текущий | При росте нагрузки | Влияние |
|--------|---------|---------------------|---------|
| RAM | 4 GB | 8-16 GB | Больше кэша в Redis |
| CPU | 2 cores | 4-8 cores | Больше запросов/сек |
| SSD | 20 GB | 50-100 GB | Больше файлов в MinIO |

### Производительность эндпоинтов

| Эндпоинт | Без кэша | С кэшем | Улучшение |
|----------|----------|---------|-----------|
| GET /api/v1/specialties | 150ms | 8ms | **18x** |
| GET /api/v1/news | 120ms | 10ms | **12x** |
| GET /api/v1/faq | 80ms | 5ms | **16x** |

## 6. Возможные вопросы и ответы

### Общие вопросы о проекте

**В: Какова главная цель проекта?**

**О:** Предоставить абитуриентам и сотрудникам колледжа удобный, быстрый и надёжный доступ к информации о специальностях, а также помочь абитуриентам выбрать подходящую специальность через профориентационный тест.

---

**В: Сколько времени заняла разработка?**

**В: Какие самые сложные технические вызовы были?**

**О:**
1. **Реализация Clean Architecture** — нужно было правильно разделить слои и не допустить утечки зависимостей
2. **Кэширование с инвалидацией** — обеспечить актуальность данных после изменений через админку
3. **Асинхронная работа с БД** — избежать блокировок и race conditions
4. **Профориентационный алгоритм** — расчёт рекомендаций на основе баллов

---

### Вопросы об архитектуре

**В: Почему выбрали Clean Architecture, а не MVC?**

**О:** MVC смешивает бизнес-логику с доступом к данным, что затрудняет тестирование и поддержку. Clean Architecture обеспечивает:
- **Изоляцию** бизнес-логики (Domain слой)
- **Тестируемость** без БД
- **Гибкость** замены компонентов

---

**В: Почему асинхронный FastAPI, а не синхронный Django?**

**О:**
- FastAPI **в 3-5 раз быстрее** для I/O операций (благодаря asyncio)
- Автоматическая **валидация через Pydantic**
- **Swagger UI** из коробки
- Меньше потребления **памяти** под нагрузкой

---

**В: Как обрабатываете конкурентные запросы?**

**О:**
- **Async/await** позволяет обрабатывать тысячи параллельных запросов без блокировки
- **Connection pooling** в PostgreSQL (через SQLAlchemy)
- **Rate limiting** защищает от злоупотреблений

---

### Вопросы о кэшировании

**В: Как работает кэширование?**

**О:** 
1. Первый запрос → данные из БД → сохранение в Redis (TTL 5-60 минут)
2. Последующие запросы → данные из Redis (5-15ms)
3. При изменении данных через админку → кэш инвалидируется (версионирование групп)

---

**В: Что происходит, если Redis падает?**

**О:** API продолжает работать — запросы идут напрямую к БД. Health check покажет статус "degraded", но сервис доступен.

---

### Вопросы о безопасности

**В: Как защищены данные пользователей?**

**О:**
- **Пароли** хэшируются bcrypt (12 rounds)
- **JWT токены** подписаны секретным ключом (HS256)
- **HTTPS** для всех эндпоинтов (Let's Encrypt)
- **CORS** ограничен конкретными origins
- **Rate limiting** защищает от brute-force

---

**В: Как предотвращаете несанкционированный доступ?**

**О:**
- JWT **scopes** (области доступа) для каждой операции
- **Суперпользователи** имеют все права, обычные пользователи — только чтение
- **Валидация** всех входных данных через Pydantic

---

### Вопросы о производительности

**В: Сколько запросов в секунду выдерживает API?**

**О:** На текущей конфигурации (2 CPU, 4GB RAM):
- **Без кэша:** ~100-200 req/sec
- **С кэшем:** ~1000+ req/sec
- **Потенциал:** При добавлении экземпляров — линейное масштабирование

---

**В: Как оптимизировали запросы к БД?**

**О:**
- **Индексы** на часто используемых полях (code, slug)
- **Пагинация** для списков (limit/offset)
- **Eager loading** для связанных данных (education_options)
- **Кэширование** для редко изменяемых данных

---

### Вопросы о бережливом производстве

**В: Как проект соответствует принципам бережливого производства?**

**О:**

| Принцип Lean | Реализация в проекте |

| **Устранение потерь** | Автоматизация ручных процессов (обновление контента) |

| **Вытягивание** | Данные запрашиваются по необходимости, не пушатся зря |
| **Совершенство** | Тесты, CI/CD, мониторинг для постоянного улучшения |
| **Ценность для клиента** | Абитуриенты быстро получают рекомендации, сотрудники экономят время |

### Технические вопросы (для технической аудитории)

**В: Как обрабатываете миграции БД в production?**

**О:** Alembic с пошаговым применением:
```bash
docker-compose exec api alembic upgrade head
```
Каждая миграция имеет `upgrade()` и `downgrade()` для отката.

---

**В: Как тестируете проект?**

**О:**
- **Unit-тесты:** Use Cases, валидация, бизнес-логика (pytest)
- **Интеграционные тесты:** Полные HTTP запросы к запущенному API
- **Покрытие:** 80%+ целевое

---

**В: Как мониторите производство?**

**О:**
- **Health check** endpoint (`/health`)
- **Логи** через Structlog (структурированные)
- **Docker stats** для метрик контейнеров
- Можно добавить **Prometheus + Grafana** для продвинутого мониторинга

## 7. Демонстрация (сценарий презентации)

### 5-минутная демонстрация

**Минута 1: Обзор**
- Открыть Swagger UI (`/docs`)
- Показать список эндпоинтов

**Минута 2: Публичный API**
- Выполнить запрос к `/api/v1/specialties`
- Показать ответ JSON

**Минута 3: Профориентационный тест**
- Выполнить POST `/api/v1/test/results`
- Показать рекомендацию специальностей

**Минута 4: Админ-панель**
- Войти через `/auth/login`
- Создать новость через админку

**Минута 5: Кэширование**
- Показать статистику кэша (`/admin/cache/stats`)
- Продемонстрировать время ответа с/без кэша
