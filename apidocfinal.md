# Приложение Б: Серверная часть API системы профориентации

## Введение

Наша команда разработала высокопроизводительное серверное API (Приложение Б) как центральную компоненту единой цифровой экосистемы профориентации и управления приёмной кампанией колледжа. Данное API служит основным источником данных для клиентских приложений (веб-сайт, мобильные приложения, административные панели) и обеспечивает надёжную интеграцию со всеми внутренними сервисами системы.

В процессе разработки мы внедрили современные архитектурные паттерны, обеспечили масштабируемость, безопасность и высокую производительность решения.

## Описание проекта

**Название проекта:** Anmicius API  
**Версия:** 1.0.0  
**Тип:** RESTful API  
**Фреймворк:** FastAPI (Python 3.11)  
**Назначение:** Предоставление программного интерфейса для доступа к данным о специальностях, профориентационном тестировании, приёмной кампании, новостях, документах и других образовательных ресурсах колледжа.

## Архитектура решения

Мы спроектировали систему на основе принципов **Clean Architecture** (Чистая архитектура), что обеспечило чёткое разделение ответственности между компонентами, независимость от фреймворков и лёгкость тестирования.

### Слои архитектуры

```
┌─────────────────────────────────────────────────────────────────┐
│                    PRESENTATION LAYER                           │
│  (FastAPI Routes, Schemas, Request/Response Handling)           │
├─────────────────────────────────────────────────────────────────┤
│                   APPLICATION LAYER                             │
│  (Use Cases, Business Logic, Dependencies)                      │
├─────────────────────────────────────────────────────────────────┤
│                     DOMAIN LAYER                                │
│  (Entities, Repository Interfaces, Business Rules)              │
├─────────────────────────────────────────────────────────────────┤
│                  INFRASTRUCTURE LAYER                           │
│  (Repositories, Database, Cache, MinIO, External Services)      │
└─────────────────────────────────────────────────────────────────┘
```

#### 1. Presentation Layer (Слой представления)

Мы реализовали этот слой с использованием FastAPI. Он отвечает за:
- Приём HTTP-запросов и валидацию входных данных
- Маршрутизацию запросов к соответствующим Use Case'ам
- Формирование HTTP-ответов в формате JSON
- Обработку исключений и ошибок
- Документирование API через Swagger UI и ReDoc

**Ключевые файлы:**
- `app/main.py` — точка входа, настройка приложения, middleware, обработчики исключений
- `app/presentation/routes.py` — публичные API эндпоинты версии v1
- `app/presentation/admin_routes.py` — административные API эндпоинты (аутентификация, CRUD операции)
- `app/presentation/schemas.py` — Pydantic-схемы для валидации запросов и ответов

#### 2. Application Layer (Слой приложения)

Этот слой содержит бизнес-логику приложения в виде Use Case'ов. Каждый Use Case инкапсулирует конкретный сценарий использования системы.

**Преимущества такого подхода:**
- Изоляция бизнес-логики от фреймворков и внешних зависимостей
- Возможность повторного использования логики в разных контекстах
- Упрощение тестирования бизнес-правил

**Ключевые файлы:**
- `app/application/use_cases.py` — Use Case'ы для публичных операций (получение специальностей, новостей, тестов и т.д.)
- `app/application/auth_use_cases.py` — Use Case'ы для аутентификации и управления пользователями
- `app/application/dependencies.py` — зависимости для внедрения (получение текущего пользователя, проверка прав)

#### 3. Domain Layer (Доменный слой)

Ядро системы, содержащее бизнес-сущности и интерфейсы репозиториев. Этот слой не зависит от других слоёв.

**Ключевые файлы:**
- `app/domain/models.py` — доменные модели (Specialty, News, TestQuestion, User и др.)
- `app/domain/repositories.py` — интерфейсы репозиториев (ISpecialtyRepository, INewsRepository и др.)

#### 4. Infrastructure Layer (Слой инфраструктуры)

Реализация технических деталей: работа с базой данных, кэшем, файловым хранилищем.

**Ключевые файлы:**
- `app/infrastructure/database.py` — асинхронное подключение к PostgreSQL, управление сессиями
- `app/infrastructure/cache.py` — сервис кэширования на базе Redis
- `app/infrastructure/minio_service.py` — работа с объектным хранилищем MinIO (S3-совместимое)
- `app/infrastructure/repositories.py` — реализации репозиториев для работы с данными
- `app/infrastructure/models.py` — SQLAlchemy-модели для маппинга на таблицы БД

## Функциональные возможности

Мы реализовали следующие ключевые модули и функции:

### 1. Модуль аутентификации и авторизации

**Возможности:**
- Регистрация новых пользователей (администраторов)
- Вход по логину и паролю с выдачей JWT-токенов (access + refresh)
- Обновление access-токена через refresh-токен
- Выход из системы с инвалидацией токенов
- Получение информации о текущем пользователе
- Ролевая модель: обычный пользователь и суперпользователь
- OAuth2 Flow для авторизации через Swagger UI

**Endpoints:**
- `POST /auth/register` — регистрация
- `POST /auth/login` — вход
- `POST /auth/login/oauth` — OAuth2 вход для Swagger
- `POST /auth/refresh` — обновление токена
- `POST /auth/logout` — выход
- `GET /auth/me` — текущий пользователь

### 2. Модуль специальностей

**Возможности:**
- Получение списка специальностей с пагинацией, поиском и фильтрацией
- Детальная информация о специальности (код, название, описание, экзамены, места, сроки обучения)
- Фильтрация по форме обучения (бюджет/платное)
- Фильтрация популярных специальностей
- CRUD-операции для администраторов
- Кэширование ответов для повышения производительности

**Endpoints (публичные):**
- `GET /api/v1/specialties` — список специальностей
- `GET /api/v1/specialties/{code}` — детали специальности
- `GET /api/v1/specialties/{code}/facts` — факты по специальности

**Endpoints (административные):**
- `GET /admin/specialties` — список для админки
- `POST /admin/specialties` — создание
- `GET /admin/specialties/{id}` — получение по ID
- `PUT /admin/specialties/{id}` — обновление
- `DELETE /admin/specialties/{id}` — удаление

### 3. Модуль интересных фактов

**Возможности:**
- Получение списка фактов для специальности (только заголовки)
- Получение полного содержимого факта с изображениями
- Привязка фактов к специальностям
- CRUD-операции для администраторов

**Endpoints (публичные):**
- `GET /api/v1/facts/{fact_id}` — детали факта

### 4. Модуль новостей

**Возможности:**
- Лента новостей с пагинацией и поиском
- Детальная страница новости с галереей изображений
- Подсчёт просмотров
- Автогенерация slug для URL
- CRUD-операции для администраторов

**Endpoints (публичные):**
- `GET /api/v1/news` — список новостей
- `GET /api/v1/news/{slug}` — детали новости

### 5. Модуль профориентационного тестирования

**Возможности:**
- Получение списка вопросов теста
- Отправка ответов пользователем
- Автоматический расчёт рекомендации на основе ответов
- Возврат рекомендованных специальностей с мотивацией
- Валидация ответов на стороне сервера
- Поддержка одиночного и множественного выбора ответов

**Endpoints (публичные):**
- `GET /api/v1/test/questions` — вопросы теста
- `POST /api/v1/test/results` — отправка ответов и получение результата

### 6. Модуль приёмной кампании

**Возможности:**
- Информация о приёмной кампании на текущий год
- Список специальностей с проходными баллами и количеством мест
- Способы подачи документов (онлайн, лично, почтой)
- Важные даты (начала и окончания приёма, публикации списков)
- Фильтрация по году поступления
- CRUD-операции для администраторов

**Endpoints (публичные):**
- `GET /api/v1/admission` — информация о приёмной кампании

### 7. Модуль FAQ (Часто задаваемые вопросы)

**Возможности:**
- Список вопросов с ответами
- Категоризация вопросов
- Возможность прикрепления изображений и документов
- Фильтрация по категории
- Флаг отображения в разделе приёмной кампании
- CRUD-операции для администраторов

**Endpoints (публичные):**
- `GET /api/v1/faq` — список FAQ

### 8. Модуль документов

**Возможности:**
- Каталог документов для скачивания
- Категоризация документов
- Информация о размере файла
- Прикреплённые изображения
- CRUD-операции для администраторов

**Endpoints (публичные):**
- `GET /api/v1/documents` — список документов

### 9. Модуль галереи

**Возможности:**
- Общая фотогалерея колледжа
- Категоризация изображений
- Превью (thumbnail) и полноразмерные изображения
- Метаданные: дата съёмки, подпись, alt-текст
- CRUD-операции для администраторов

**Endpoints (публичные):**
- `GET /api/v1/images` — галерея изображений

### 10. Модуль информации о колледже

**Возможности:**
- Общая информация о колледже (название, описание, изображения)
- Кэширование для быстрого доступа
- CRUD-операции для администраторов

**Endpoints (публичные):**
- `GET /api/v1/about` — информация о колледже

### 11. Модуль загрузки файлов

**Возможности:**
- Загрузка изображений (JPG, PNG, GIF, WebP) до 10 MB
- Загрузка документов (PDF, DOC, DOCX) до 50 MB
- Генерация уникальных имён файлов
- Сохранение в MinIO с публичным доступом
- Валидация MIME-типов и размеров
- Интеграция с другими модулями (новости, специальности, галерея)

**Endpoints (административные):**
- `POST /admin/upload/image` — загрузка изображения
- `POST /admin/upload/document` — загрузка документа

### 12. Модуль управления пользователями (Администрирование)

**Возможности:**
- Просмотр списка всех пользователей с пагинацией
- Получение информации о конкретном пользователе
- Редактирование данных пользователя (email, username, статус активности, права суперпользователя)
- Удаление пользователей
- Проверка прав доступа через scopes

**Endpoints (административные):**
- `GET /admin/users` — список пользователей
- `GET /admin/users/{user_id}` — пользователь по ID
- `PATCH /admin/users/{user_id}` — обновление
- `DELETE /admin/users/{user_id}` — удаление

### 13. Модуль управления кэшем (Администрирование)

**Возможности:**
- Принудительная очистка всего кэша
- Очистка кэша по группам (public, admin)
- Мониторинг состояния кэша

**Endpoints (административные):**
- `DELETE /admin/cache` — очистка всего кэша
- `DELETE /admin/cache/group/{group_name}` — очистка группы

| GET | `/admin/admission` | Приёмная кампания (админ) | Да | - |
| POST | `/admin/admission` | Создание записи | Да | - |
| PUT | `/admin/admission/{year}` | Обновление записи | Да | - |
| DELETE | `/admin/admission/{year}` | Удаление записи | Да | - |

## Модель данных

### Основные сущности базы данных

#### Specialty (Специальности)
- `id` (Primary Key)
- `code` (VARCHAR, unique) — код специальности (например, "09.02.07")
- `name` (VARCHAR) — полное название
- `short_description` (TEXT) — краткое описание
- `description` (JSONB) — подробное описание (список абзацев)
- `exams` (JSONB) — список требуемых экзаменов
- `images` (JSONB) — массив изображений
- `documents` (JSONB) — массив связанных документов
- `created_at`, `updated_at` (TIMESTAMP)

#### EducationOption (Уровни образования)
- `id` (Primary Key)
- `specialty_id` (Foreign Key → Specialty)
- `education_level` (VARCHAR) — уровень образования
- `duration` (VARCHAR) — срок обучения
- `budget_places` (INTEGER) — количество бюджетных мест
- `paid_places` (INTEGER) — количество платных мест

#### News (Новости)
- `id` (Primary Key)
- `title` (VARCHAR)
- `slug` (VARCHAR, unique) — URL-friendly идентификатор
- `preview_text` (TEXT)
- `content` (JSONB) — содержимое новости
- `preview_image` (VARCHAR) — URL превью
- `gallery` (JSONB) — галерея изображений
- `published_at` (TIMESTAMP)
- `views` (INTEGER) — счётчик просмотров

#### TestQuestion (Вопросы теста)
- `id` (Primary Key)
- `text` (TEXT) — текст вопроса
- `options` (JSONB) — варианты ответов
- `answer_scores` (JSONB) — баллы для каждой специальности
- `image_url` (VARCHAR) — изображение вопроса
- `documents` (JSONB) — прикреплённые документы

#### User (Пользователи)
- `id` (Primary Key)
- `email` (VARCHAR, unique)
- `username` (VARCHAR, unique)
- `hashed_password` (VARCHAR)
- `is_active` (BOOLEAN)
- `is_superuser` (BOOLEAN)
- `created_at`, `updated_at` (TIMESTAMP)

## Работа с базой данных и миграции

## Кэширование

Мы внедрили многоуровневую систему кэширования на базе **Redis** для значительного повышения производительности API.

### Стратегия кэширования:

1. **Публичные данные** (группа `public`):
   - Информация о колледже: TTL = 3600 сек (1 час)
   - Список специальностей: TTL = 300 сек (5 минут)
   - Детали специальности: TTL = 600 сек (10 минут)
   - Новости: TTL = 300 сек (5 минут)
   - Детали новости: TTL = 600 сек (10 минут)
   - FAQ, документы, факты: TTL = 600 сек (10 минут)

2. **Генерация ключей кэша**:
   Ключи формируются динамически на основе параметров запроса:
   ```
   specialties:p{page}:l{limit}:s{search}:f{form}:pop
   news:p{page}:l{limit}:s{search}
   faq:{category}
   ```

3. **Инвалидация кэша**:
   - Автоматическая при изменении данных через административные API
   - Принудительная через endpoints `/admin/cache`

4. **Отказоустойчивость**:
   - При недоступности Redis API продолжает работу без кэширования
   - Логирование ошибок кэша без прерывания запроса

## Безопасность

Мы внедрили комплексный подход к безопасности API:

### 1. Аутентификация и авторизация
- **JWT-токены** (JSON Web Tokens) с алгоритмом HS256
- **Access-токен**: время жизни 30 минут
- **Refresh-токен**: время жизни 7 дней
- **OAuth2 Password Flow** для интеграции со Swagger UI
- **Scopes-based авторизация**: granular доступ к операциям (users:read, specialties:write и т.д.)

### 2. Хранение паролей
- Хеширование через **bcrypt** с cost factor = 12
- Использование библиотеки `passlib` для удобной работы

### 3. Валидация входных данных
- Строгая валидация через **Pydantic**-схемы
- Проверка типов, диапазонов, форматов
- Санитизация пользовательского ввода

### 4. CORS (Cross-Origin Resource Sharing)
- Настройка разрешённых источников через ENV-переменные
- Разрешены только доверенные домены:
  - `https://anmicius.ru`
  - `https://api.anmicius.ru`
  - `https://admin.anmicius.ru`
  - `https://minio.anmicius.ru`

### 5. Rate Limiting (Ограничение частоты запросов)
- Реализовано через библиотеку **SlowAPI**
- Лимиты для критичных endpoints:
  - `/auth/register`: 10 запросов/минуту, 30/час
  - `/auth/login`: 10 запросов/минуту, 30/час
  - `/auth/refresh`: 30 запросов/минуту, 100/час
- Ответ при превышении: HTTP 429 Too Many Requests

### 6. Требования к паролям
- Минимальная длина: 8 символов
- Обязательно: хотя бы одна цифра
- Обязательно: хотя бы одна буква верхнего регистра
- Обязательно: хотя бы одна буква нижнего регистра

### 7. Защита от распространённых уязвимостей
- SQL Injection: предотвращено через параметризированные запросы SQLAlchemy
- XSS: санитизация входных данных, Content-Type заголовки
- CSRF: использование токенов для state-changing операций
